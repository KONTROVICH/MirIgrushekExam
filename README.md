## 🧸 Описание предметной области: ООО «МирИгрушек»

ООО «МирИгрушек» — это специализированный магазин по продаже игрушек. Для оптимизации бизнес-процессов компании критически важно постоянно получать актуальную информацию об остатках товаров на складах. В рамках выполнения задания демонстрационного экзамена необходимо разработать основные модули информационной системы, обеспечивающие строгий контроль доступа, учет номенклатуры и управление заказами.

### Разграничение прав доступа по ролям:
1. **Гость:** Имеет доступ только к экрану просмотра товаров. Функции фильтрации, сортировки и поиска для данной роли полностью заблокированы.
2. **Авторизованный клиент:** Обладает аналогичными гостю правами — может исключительно просматривать товары без возможности использования инструментов поиска, фильтрации и сортировки.
3. **Менеджер:** Имеет право просматривать список товаров с применением фильтрации, сортировки и поиска, а также просматривать список существующих заказов (без возможности их изменения).
4. **Администратор:** Обладает полными правами управления системой (CRUD). Может просматривать, добавлять, редактировать и удалять информацию как по товарам, так и по заказам.

### Модуль № 1: Разработка базы данных средствами СУБД
* **Продолжительность выполнения:** 50 минут.

#### Требования к проектированию:
1. Создать базу данных в выбранной СУБД на основе описания предметной области и предоставленных файлов заказчика.
2. Структура БД должна строго соответствовать **третьей нормальной форме (3НФ)**.
3. Обеспечить полную ссылочную целостность данных. Настроить первичные (PK) и внешние (FK) ключи, соблюдая логичную и согласованную схему именования сущностей и атрибутов.
4. Выполнить подготовку и импорт данных из файлов, предоставленных заказчиком (файлы с пометкой *import* в ресурсах).
5. **Финальные артефакты:** * Сформировать скрипт создания структуры и наполнения БД (`.sql`) или файл конфигурации с данными (`.dt`) для платформы 1С.
   * Получить ER-диаграмму средствами СУБД или специализированного ПО. Диаграмма представляется в формате **PDF** и должна содержать все таблицы, связи, атрибуты и ключи (типами данных на данном этапе можно пренебречь).

### Модуль № 2: Разработка алгоритма и создание приложения
* **Продолжительность выполнения:** 40 минут.

#### Требования к коду и архитектуре:
1. **Алгоритм:** Сформировать алгоритм разработки ПО и оформить его в виде блок-схемы в строгом соответствии со стандартом **ГОСТ 19.701-90**. Документ сохранить в формате **PDF**.
2. **Оформление кода:** Идентификаторы должны отражать их назначение. Соответствие стилю именования зависит от технологии: `CamelCase` (для C# и Java), `snake_case` (для Python), либо стандарты фирмы 1С.
3. **Ограничение строк:** Допустимо использование не более одной команды на строку кода.
4. **Связь с БД:** Подключить созданную базу данных к приложению. Список товаров должен динамически отображать актуальную информацию из БД.

#### Интерфейс авторизации:
1. При запуске приложения окно входа — это первый экран, который видит пользователь.
2. Пользователю предлагается ввести логин и пароль (проверяются по БД) либо нажать кнопку для перехода к просмотру товаров в роли гостя.
3. Доступ к остальным закрытым модулям открывается только после успешной аутентификации.
4. В любой учетной записи в **правом верхнем углу экрана** должны отображаться ФИО текущего пользователя.
5. Из любого аккаунта должна быть доступна функция безопасного выхода (кнопка разлогинивания) с возвратом на исходное окно входа.

#### Отображение списка товаров:
Каждый товар в списке (доступном для всех 4 ролей) должен отображаться в виде карточки/строки, содержащей следующие данные:
* Фото товара (если изображение отсутствует, выводится картинка-заглушка `picture.png` из ресурсов).
* Наименование товара и его Категория.
* Описание товара, Производитель и Поставщик.
* Базовая цена, Единица измерения и Количество на складе.
* Действующая скидка.

#### Правила условного форматирования интерфейса:
* Если действующая скидка на товар **строго превышает 17%**, фон карточки/строки этого товара окрашивается в цвет `#FFDEAD`.
* Если у товара снижена цена, его базовая (старая) цена должна быть **перечеркнута и отображена красным шрифтом**, а рядом выводится актуальная итоговая стоимость черным шрифтом.
* Если товара **нет на складе** (количество равно 0), вся строка/карточка выделяется **голубым цветом**.

*Примечание к модулю:* Необходимо провести полную отладку. Приложение должно работать стабильно, без аварийных завершений. Результаты проверок (скриншоты корректной работы) вкладываются в отчетный документ формата **DOCX**.

### Модуль № 3: Разработка последовательного интерфейса и CRUD товаров
* **Продолжительность выполнения:** 1 час 30 минут.

#### Навигация и обработка ошибок:
1. Реализовать последовательный пользовательский интерфейс, позволяющий свободно перемещаться между экранами (включая переходы назад, например, по кнопке «Назад»). Каждое окно обязано иметь корректный заголовок.
2. **Исключения:** Реализовать обработку исключительных ситуаций. Приложение должно уведомлять пользователя об ошибках ввода, запрещенных действиях или предупреждать о неотвратимых операциях (например, удалении). Окна сообщений (ошибка, предупреждение, информация) должны содержать пиктограмму, понятный заголовок и информативный текст с порядком действий для исправления. Допускается использование визуальных подсказок на формах.
3. **Документирование:** Использовать комментарии для пояснения сложных или неочевидных фрагментов кода.

#### Поиск, фильтрация и сортировка:
*Данный функционал доступен исключительно Менеджеру и Администратору.*
1. **Поиск:** Текстовая поисковая строка осуществляет фильтрацию одновременно по всем текстовым полям (мультиатрибутивный поиск).
2. **Сортировка:** Возможность упорядочивания данных по возрастанию и по убыванию по двум критериям: `Цена` и `Количество на складе`.
3. **Фильтрация:** Выборка данных по полю `Поставщик`. Первым элементом выпадающего списка должен быть пункт **"Все поставщики"**, выбор которого полностью сбрасывает текущие настройки фильтра.
4. **Реальное время (Динамичность):** Поиск, сортировка и фильтрация должны срабатывать автоматически в реальном времени при изменении условий, без необходимости нажатия дополнительных кнопок («Найти», «Применить»).
5. **Совместное применение:** Поиск и фильтрация работают совместно, сохраняя при этом ранее выбранные параметры сортировки.

#### Управление товарами (Функционал Администратора):
1. Администратор имеет монопольное право на добавление, редактирование и удаление товаров.
2. Форма добавления/редактирования должна открываться в **новом отдельном окне**. Во избежание конфликтов редактирования, в системе должно быть заблокировано открытие более одного окна редактирования одновременно.
3. При редактировании все существующие поля объекта из БД автоматически подгружаются в форму.
4. **Логика ID:** Поле `ID товара` при добавлении нового объекта на форме скрыто и рассчитывается автоматически как `+1` к максимальному ID в базе данных. При редактировании существующего товара поле `ID` отображается в режиме "только для чтения".
5. **Валидация данных:** Стоимость товара может включать сотые части (копейки), но не может быть отрицательной. Минимальное количество товара на складе также не может быть меньше 0. Категории и поставщики выбираются из выпадающих списков.
6. **Работа с графикой:** Администратор может загружать и заменять фото товара. Установлено жесткое ограничение на размер фото — **300х200 пикселей**. Картинка физически сохраняется в папку с приложением, а в БД записывается путь к ней. При замене изображения старый файл должен физически удаляться с диска.
7. **Бизнес-логика удаления:** Запрещено удалять товар со склада, если он фигурирует хотя бы в одном существующем заказе.
8. После совершения любых CRUD-операций (добавление, изменение, удаление) список товаров на главной форме должен обновляться мгновенно.

### Модуль № 4: Работа с заказами
* **Продолжительность выполнения:** 1 час 00 минут.

#### Требования к модулю:
1. В интерфейсы ролей Менеджера и Администратора должна быть добавлена кнопка **«Заказы»**.
2. Окно списка заказов выводит информацию по следующим полям:
   * Артикул заказа
   * Статус заказа
   * Адрес пункта выдачи (текст)
   * Дата заказа
   * Дата доставки / выдачи
3. **Управление заказами:** Возможность добавления и редактирования заказов доступна только Администратору.
4. Операции добавления нового заказа или изменения параметров текущего осуществляются в **новом отдельном окне (форме)**. 
5. Переход к форме редактирования происходит при нажатии на конкретный заказ в списке, а к форме добавления — при нажатии на кнопку «Добавить заказ».
6. На форме должны быть предусмотрены следующие поля: артикул, статус заказа (выпадающий список), адрес пункта выдачи, дата заказа, дата выдачи.
7. Реализовать возможность удаления заказа Администратором.
8. После любых изменений (редактирование/добавление/удаление) данные в окне списка заказов должны автоматически обновляться.
